# Publishing the Ateam iOS app (TestFlight → App Store)

The mobile app (`apps/mobile`) is an Expo SDK 52 **bare** app: `app.json` is the
source of truth and the native `ios/` project is **generated by `expo prebuild`**
(it's git-ignored — never hand-edit `ios/` and expect it to stick). The vendored
SwiftTerm terminal lives in the tracked `modules/expo-swiftterm` Expo module and
**re-autolinks on every prebuild**, so regenerating `ios/` is safe.

## Distribution strategy

Ship on **internal TestFlight first**, then widen only once onboarding is solved.
Why (the full reasoning is in the `/scalable` ruling in chat, condensed here):

- **Internal TestFlight** — up to 100 team testers, **no App Review**, installs
  immediately. This is OTA to your phone with zero gatekeeping. Start here.
- **External TestFlight** (up to 10k) and a **public App Store listing** both trigger
  **Beta/App Review**, which requires a reviewer to reach a *working* backend. Ateam
  needs a proprietary `ateam daemon` a reviewer can't stand up, so both need a
  **review-only demo box** (see below). Same uploaded build, more rings flipped on —
  no rebuild, no rework.
- **TestFlight builds expire after 90 days.** Your release cadence already beats that;
  automate the upload (EAS or CI) rather than hand-cranking Xcode each quarter.

## One-time setup

1. **Apple Developer Program** — team **`X2VZX44YM2`** (the same Clawnify publishing
   team used for the macOS releases; your macOS Developer ID proves it's enrolled).
2. **App Store Connect** → **My Apps → +** → new app:
   - Platform: iOS · Bundle ID: **`com.clawnify.ateam`** · SKU: `ateam-mobile`.
   - The bundle ID must first be registered as an App ID in **Certificates,
     Identifiers & Profiles → Identifiers** (Explicit, no capabilities) under team
     `X2VZX44YM2`, then it appears in the New App dialog's dropdown. It's permanent
     once the app record exists.
3. **Signing** — easiest is Xcode's *Automatically manage signing* with team
   `X2VZX44YM2`; it provisions the Apple Distribution cert + App Store profile.

## Per-release checklist

`app.json` already carries the release identity:

| Field | Value | Becomes (native) |
| --- | --- | --- |
| `expo.name` | `Ateam Go` | `CFBundleDisplayName` (name under the icon; matches the App Store name) |
| `expo.version` | `1.0.0` | `CFBundleShortVersionString` (marketing version) |
| `expo.ios.buildNumber` | `1` | `CFBundleVersion` (**must increase every upload**) |
| `expo.ios.infoPlist.ITSAppUsesNonExemptEncryption` | `false` | skips the export-compliance prompt on every upload¹ |

**Bump `ios.buildNumber` on every TestFlight upload** (App Store Connect rejects a
duplicate `version`+`buildNumber` pair). Keep `version` until you cut a user-visible
release.

¹ Declares the app uses only standard/exempt encryption (OS-provided TLS/SSH/WebSocket —
no custom crypto). Correct for Ateam; revisit if you ever add proprietary encryption.

## Build & upload — Route A: local Xcode archive (recommended first)

Reuses the exact toolchain that already produces working device builds (SwiftTerm
module, SDK-52 / Xcode-16.2 pinning):

```bash
cd apps/mobile
npx expo prebuild -p ios            # regenerate ios/ from app.json (SwiftTerm re-autolinks)

open ios/mobile.xcworkspace
#  in Xcode:
#   1. select scheme "mobile", destination "Any iOS Device (arm64)"
#   2. Signing & Capabilities → team X2VZX44YM2, automatic signing
#   3. Product ▸ Archive   (Release)
#   4. Organizer opens → Distribute App ▸ App Store Connect ▸ Upload
```

Headless equivalent (for later CI):

```bash
xcodebuild -workspace ios/mobile.xcworkspace -scheme mobile \
  -configuration Release -destination 'generic/platform=iOS' \
  -archivePath build/Ateam.xcarchive archive
xcodebuild -exportArchive -archivePath build/Ateam.xcarchive \
  -exportOptionsPlist ios/ExportOptions.plist -exportPath build/export
xcrun altool --upload-app -f build/export/mobile.ipa -t ios \
  --apiKey "$ASC_KEY_ID" --apiIssuer "$ASC_ISSUER_ID"   # App Store Connect API key
```

## Build & upload — Route B: EAS (scalable follow-up)

Cloud-reproducible and CI-friendly, but needs an `eas.json` + credentials setup that
must account for the bun workspace + vendored native module. Do this **after** a first
successful manual submission, not as the way in:

```bash
bunx eas-cli build   -p ios --profile production
bunx eas-cli submit  -p ios --latest
```

## After upload

1. App Store Connect processes the build (~5–15 min).
2. **TestFlight** tab → add yourself under **Internal Testing** (no review) → install
   the TestFlight app on your phone. Every later upload is OTA to all testers.
3. Each build is usable for **90 days**; ship a fresh one before it lapses.

## Going wider (external testers / public listing)

Both need a reviewer to exercise a live backend. Provide it via **App Review
Information → Demo account**:

- Run a **review-only `ateam daemon`** on a box you control, with a demo project and
  baked-in demo connection details. Keep it live during review (*"backend services
  must be enabled and live"* — Apple guideline 2.1).
- This stays review-only, so it doesn't compromise Ateam's **local-first** model —
  real users still self-host their own box (see [online-ateam.md](./online-ateam.md)).

A public listing additionally needs: a real app name that isn't taken, a privacy
policy URL, an age rating, and screenshots. The app icon (`assets/icon.png`) is
already 1024×1024 with no alpha — App Store compliant.
